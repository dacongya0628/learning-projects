# AircraftWars
> 一个使用原生 Canvas 和定时循环实现的 H5 简化版飞机大战学习项目。

这个项目用最基础的前端技术组合搭了一个可运行的 2D 小游戏原型：页面加载后创建战机、敌机和子弹对象，通过画布逐帧重绘来完成移动、碰撞、计分和结算逻辑。它不是工程化游戏项目，而是偏向“把小游戏核心机制手写出来”的练习仓库，适合用来理解原生 Canvas 游戏循环和对象拆分方式。

## 🎯 学习目标

- 理解原生 `canvas` 在小游戏中的基础绘制方式
- 练习将战机、敌机、子弹拆分为独立对象管理
- 学习通过定时器和逐帧刷新驱动游戏循环
- 体验碰撞检测、计数更新和结算判断的基础实现
- 熟悉移动端触摸控制与简单键盘控制的接入方式

## ⚙️ 设计要点

### 1. 页面结构是“画布 + DOM 状态层”

`index.html` 里并不只有 `canvas`，还叠加了开始按钮、计数面板、结束弹层和底部信息栏。也就是说，游戏主体绘制在画布中，游戏状态展示和操作入口则交给普通 DOM 元素完成，这种方式对学习项目很直接，也便于快速调整界面。

### 2. 游戏对象按职责拆成 3 类

- `Plane` 负责玩家战机的移动、绘制和发射逻辑
- `Fire` 负责子弹上移、渲染与命中敌机
- `Enemy` 负责敌机下落、渲染与逃脱统计

这种拆分比较适合练习“每种游戏对象各管自己的更新和绘制”，但当前对象之间仍然通过全局 `Main` 共享状态，耦合度不低。

### 3. 主循环和对象循环并存

项目同时用了两种时间驱动方式：

- `Main.timer` 基于 `requestAnimationFrame` 周期执行清屏和结束判定
- `Plane` 通过 `setInterval` 持续创建子弹和敌机
- `Fire` / `Enemy` / `Plane` 自身又各自维护一个 `Timer.TimeProcess`

这让项目比较容易跑起来，但后续继续扩展时，时间源会变得分散，不利于统一调速、暂停和排查性能问题。

### 4. 结算条件简单明确

当前唯一的失败条件是逃脱敌机达到 3 架，玩家战机没有生命值扣减、无敌机撞机判定、无关卡系统，也没有分数难度曲线之外的更多成长机制。这个范围控制让代码更适合作为入门练手项目。

## 🧭 开发指北

### 核心模块

1. **`js/main.js`**  
   负责初始化画布、加载图片资源、创建战机实例，并在开始游戏后切换 UI 状态、监听触摸操作、检查游戏是否结束。

2. **`js/plane.js`**  
   是玩家侧的主要控制器。这里不仅处理战机位置更新，还负责开启两个间隔任务：一个生成子弹，一个生成敌机。

3. **`js/fire.js`**  
   子弹每帧向上移动，同时遍历 `Main.enemyArr` 做简单矩形范围命中判断；命中后会同时删除子弹和敌机，并更新击毁计数。

4. **`js/enemy.js`**  
   敌机每帧向下移动，超出底部后计入逃脱数；随着击毁数增加，敌机速度会通过全局变量 `addNum` 逐步提升。

5. **`js/common.js`**  
   放了几个基础能力：简易 DOM 查询、图片预加载、`Frame` 结构和 `Timer.TimeProcess` 时间循环工具。这里更像整个项目的轻量运行时基础设施。

### 继续扩展时优先关注的点

- 把子弹和敌机的索引管理从“创建时写死”改成实时计算或对象引用删除，避免数组删除后索引失真
- 将碰撞检测统一收敛到主循环，减少对象内部直接互相操作全局数组的耦合
- 明确桌面端控制方式，例如在开始游戏后主动聚焦可接收键盘事件的容器
- 为战机加入生命值、敌机撞击判定、暂停/继续、难度分级等更完整的玩法机制
- 处理窗口尺寸变化时的画布重设与重绘，当前实现默认只按首次进入时的视口运行

## 📚 技术收获

- 小游戏原型并不一定需要框架，原生前端技术就足够完成基础交互闭环
- 把绘制、更新、碰撞、计数拆开，是理解游戏循环的关键一步
- 资源预加载、对象数组管理、定时器协作，是这类项目最容易暴露维护问题的地方
- 当项目继续长大后，统一时间轴和统一状态管理会比“先跑起来”更重要
